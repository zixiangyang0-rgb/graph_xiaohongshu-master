## 一、项目概览与运行

### 1.1 项目是什么

一个基于 **LangGraph** + **FastAPI** + **PostgreSQL** 构建的 AI 内容生成工作流。

用户输入主题方向（如"AI技术"），AI 自动完成以下完整流程：

```
用户输入主题方向（如"AI技术"）
        │
        ▼
┌──────────────────┐
│  plan_topics     │  ← AI 生成 3-5 个选题
│  (选题生成节点)   │
└──────────────────┘
        │
        ▼
┌──────────────────┐
│  ⏸ 人工选题      │  ← 工作流暂停，等用户选择
└──────────────────┘
        │
        ▼
┌──────────────────┐
│  write_draft     │  ← AI 根据选题撰写文章（流式输出）
│  (文章撰写节点)   │
└──────────────────┘
        │
        ▼
┌──────────────────┐
│  ⏸ 人工审稿      │  ← 工作流暂停，等用户审核
└──────────────────┘
        │
        ▼
┌──────────────────┐
│  extract_visuals │  ← 从文章提取配图要点
│  generate_images │  ← 并行调用 Gemini 生成多张配图
└──────────────────┘
        │
        ▼
     工作流完成
```

### 1.2 核心设计理念

**Human-in-the-loop（人在回路）**：在选题和审稿环节引入人工决策，而非纯自动化。AI 负责生成内容，但人做最终把控。

**状态持久化**：通过 PostgreSQL + LangGraph Checkpointer 保存工作流状态，支持断点续传、故障恢复、多线程并行。

**流式输出**：使用 Server-Sent Events（SSE）实现实时流式响应，用户可以看到文章逐字生成的过程。

**PII 匿名化**：在日志和 LLM 输入输出中自动脱敏敏感信息（邮箱、手机号、身份证等）。

### 1.3 技术栈分层

| 层级 | 技术 | 说明 |
|------|------|------|
| API 层 | FastAPI | 异步 Web 框架，提供 REST API |
| 工作流层 | LangGraph 1.0+ | 工作流编排引擎，支持 interrupt/Command/Checkpointer |
| AI 层 | LangChain + 火山引擎 Doubao LLM | 调用大语言模型 |
| 图像层 | Gemini Image API | 生成配图 |
| 数据层 | SQLAlchemy 异步 ORM + PostgreSQL | 存储用户数据和 LangGraph 检查点 |
| 安全层 | JWT 认证 + Argon2 密码哈希 + PII 脱敏 | 认证和安全 |

#### 环境准备

确保已安装：
- **Python 3.10+**
- **PostgreSQL**（已运行）

8. 新开终端，启动前端：`cd frontend && npm install && npm run dev`

## 五、LangGraph 工作流

### 5.1 什么是 LangGraph

LangGraph 是 LangChain 生态中的工作流编排引擎，它的核心思想是：**用图（Graph）的形式描述业务流程**。

```
图 = 节点（Node）+ 边（Edge）
节点 = 一个个具体的步骤
边 = 步骤之间的连接关系
```

类比：就像流程图，每个方框是一个节点，箭头是边。

### 5.2 状态定义

LangGraph 用一个 **State**（状态）对象在节点之间传递数据。每个节点接收当前状态，做处理，然后返回要更新的字段。

工作流核心状态字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `topic_direction` | str | 用户输入的主题方向 |
| `topics` | list | AI 生成的选题列表 |
| `selected_topic` | str | 用户选中的选题 |
| `article_content` | str | 生成的文章内容 |
| `review_result` | str | 审核结果："approved" 或 "rejected" |
| `feedback` | str | 驳回时的反馈意见 |
| `visual_points` | list | 从文章提取的配图要点 |
| `generated_images` | list | 生成的图片路径 |
| `thread_id` | str | 线程唯一标识 |
| `user_id` | int | 用户 ID |

### 5.3 工作流组装

LangGraph 的组装流程：

```
1. 创建图：StateGraph(WorkflowState)
2. 添加节点：graph.add_node("plan_topics", plan_topics_node)
3. 添加边：graph.add_edge(START, "plan_topics")
4. 编译：compiled_graph = graph.compile(checkpointer=...)
```

边决定了工作流的走向：
- `START` → 起点
- `END` → 终点
- 节点之间用 `add_edge` 连接

### 5.4 节点详解

#### plan_topics — 选题生成节点

负责根据用户输入的主题方向，调用 LLM 生成 3-5 个候选选题。LLM 使用 JSON Mode 输出，保证格式一致。

#### write_draft — 文章撰写节点

负责根据选定的选题撰写文章。支持流式输出，文章内容通过 SSE 实时推送给前端。还支持 revision 循环：如果用户驳回，该节点会结合反馈意见重新生成。

#### human_review — 人工审核节点

这是最关键的节点。使用了 LangGraph 的 **interrupt** 机制：

```
interrupt = 暂停工作流，等待外部信号恢复
```

当工作流执行到这个节点时：
1. 工作流暂停，状态自动保存到 PostgreSQL
2. 前端检测到中断信息，显示操作界面（选题选择/审核按钮）
3. 用户操作后，调用 `/resume` 接口
4. 工作流从中断点恢复，继续执行

#### extract_visuals — 配图要点提取节点

从已通过审核的文章内容中，提取适合配图的视觉描述（3-5 个）。

#### generate_images — 配图生成节点

并行调用 Gemini Image API，同时生成多张配图。

### 5.5 interrupt（中断）原理详解

interrupt 是 LangGraph 最强大的特性之一，它的意义在于：

**没有 interrupt 时**：工作流从启动到结束一气呵成，中间无法插入人工决策。

**有了 interrupt 后**：工作流可以"停下来等一等"，等人做完决定再继续。

```
用户点击"启动工作流"
        ↓
plan_topics 节点执行 → 生成选题
        ↓
遇到 human_review 节点的 interrupt → 工作流暂停
        ↓
状态自动保存到 PostgreSQL（断点续传的保障）
        ↓
前端收到 interrupt_info → 显示选题列表
        ↓
用户选择一个选题，点击"确定"
        ↓
调用 /resume 接口 → 工作流恢复
        ↓
write_draft 节点执行 → 生成文章
        ↓
遇到第二个 interrupt → 暂停
        ↓
前端显示"审核通过/驳回"按钮
        ↓
用户点击"通过"
        ↓
调用 /resume 接口 → 工作流恢复
        ↓
extract_visuals → generate_images → 完成
```

### 5.6 子图：topic_selection.py

选题选择子图负责在多个 AI 生成的选题中做选择，包括：
- 显示选题列表供用户选择
- 支持用户自定义选题
- 处理选题冲突

## 七、安全与中间件

### 7.1 JWT 认证

路径：`app/core/security.py`

JWT（JSON Web Token）是一种无状态的认证方式。用户登录成功后，服务器签发一个 Token，客户端在后续请求中携带这个 Token，服务器验证 Token 的有效性即可确认用户身份，无需查询数据库。

### 7.2 密码哈希

使用 Argon2 算法对密码进行哈希存储。注册时哈希，登录时验证。使用专门的哈希算法而非加密，因为哈希是单向的，即使数据库泄露也无法反推原始密码。

### 7.3 中间件

路径：`app/core/middleware.py`

1. **请求日志中间件** — 记录每个请求的 method、path、status、duration
2. **LangSmith 追踪**（可选） — AI 应用的可观测性

### 7.4 LangChain 回调

路径：`app/core/callbacks.py`

在 LangChain 调用时注入 PII 脱敏回调：脱敏输入 prompt、脱敏输出 response、写入结构化日志。

## 九、JWT 鉴权：无状态登录方案

### 9.1 什么是"鉴权"？

```
鉴权 = 验证身份 = 证明"你是你"

去医院：
  你说"我是张三" → 护士说"证明给我看" → 你掏出身份证 → 护士验完：是真的，进去吧

去 API：
  你说"我是用户123" → 服务器说"证明给我看" → 你掏出 JWT → 服务器验完：是真的，处理请求
```

### 9.2 传统方式 vs JWT 方式

#### 传统方式（Session）：每次都要查档案

```
你进医院 → 护士问"你是谁" → 查医院档案室（数据库）→ 找到了，你是张三
                                          ↓
                              在档案室登记：你正在就诊
                                          ↓
                              给你一张就诊卡（session_id）
                                          ↓
每次检查都要带卡 → 护士查档案室 → 确认"张三正在就诊" → 继续
```

**问题**：
- 档案室（数据库）要一直查，很慢
- 医院大了，档案室挤满人（高并发下数据库压力大）
- 卡丢了/过期了还要重新办

#### JWT 方式：把身份证直接给你

```
你进医院 → 护士问"你是谁" → 医院查档案室（数据库）→ 找到了，你是张三
                                          ↓
                              给你一张"电子身份证"（JWT）
                                          ↓
                              上面写着：有效期到明天，持有者：张三，权限：普通门诊
                                          ↓
以后每次检查 → 你直接掏出这张电子身份证 → 护士扫一眼 → 不用再查档案室了
```

**区别**：
- 传统方式：每次请求都要查档案室（数据库）
- JWT：登录时查一次，以后直接看身份证

### 9.3 JWT 长什么样？

JWT 是一串用 `.` 分隔的字符串，分成三部分：

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.
eyJ1c2VyX2lkIjoxMjM0LCJ1c2VybmFtZSI6IumZrueUtyIsImV4cCI6MTcwOTI0MjAwMH0.
SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
        ↓                              ↓                                ↓
     Header                          Payload                       Signature
  (头部：算法信息)              (负载：用户数据)                  (签名：防伪验证)
```

#### Header（头部）

告诉别人怎么看这张证：

```json
{
  "alg": "HS256",   // 我用什么算法加密的
  "typ": "JWT"      // 我是 JWT 格式的
}
```

#### Payload（负载）

证上写的内容：

```json
{
  "user_id": 12345,
  "username": "张三",
  "role": "admin",
  "exp": 1709240200   // 什么时候过期
}
```

#### Signature（签名）

防伪码。用服务器才有的密钥，对 Header 和 Payload 一起做加密运算。如果有人改了 Payload，签名就对不上了，服务器能发现。

### 9.4 JWT 完整流程

```
第一步：登录
用户：POST /login  { username: "张三", password: "123456" }
服务器：查数据库 → 密码对 → 生成 JWT → 返回给用户

第二步：以后请求都带着 JWT
用户：GET /api/content
      Header: Authorization: Bearer eyJhbGci...

第三步：服务器验证 JWT
服务器：拿到 JWT → 验证签名 → 没过期 → 取出 user_id → 处理请求
       （全程没查数据库！）
```

### 9.5 JWT vs Session 对比

| | Session | JWT |
|---|---|---|
| 存储位置 | 服务器（内存/Redis/数据库） | 客户端（浏览器/localStorage） |
| 查数据库 | 每次请求都要查 | 只登录时查一次 |
| 服务器压力 | 高（大并发下数据库爆了） | 低（无状态） |
| 扩展性 | 难（多台服务器要共享 session） | 简单（谁都能验证 JWT） |
| 能撤回吗 | 能（删 session 就行） | 难（只能等过期） |
| 适合场景 | 需要强制登出 | 不需要实时撤回权限 |

## 十、OAuth2.0 单点登录（SSO）

### 10.1 什么是 SSO？

**场景**：一个账户登录淘宝，天猫、飞猪、优酷自动也登录了。

```
不用 SSO：
  淘宝 → 登录 → 记住了
  天猫 → 再登录一遍 → 又记一遍
  飞猪 → 再再登录一遍 → 烦死了

用了 SSO：
  淘宝 → 登录（你把账号密码给 SSO 系统）
  天猫 → 已经登录了（天猫问 SSO：这人是谁？→ SSO 说：已经验过了）
  飞猪 → 已经登录了
```

### 10.2 OAuth2.0 的四种授权方式

| 方式 | 场景 | 比喻 |
|------|------|------|
| 授权码（Authorization Code） | 网站登录第三方 | 去物业登记，物业给你一张通行卡 |
| 隐式（Implicit） | 纯前端 SPA | 直接给你钥匙，不管你身份 |
| 密码凭证（Password） | 可信 App | 你把密码直接给 App |
| 客户端凭证（Client Credentials） | 机器对机器 | 服务器之间互相验证 |

### 10.3 授权码模式流程

```
用户 → 点"用微信登录"
        ↓
    跳到微信授权页 → 用户点"授权"
        ↓
    微信回调你的网站 → 带一个 code（授权码）
        ↓
    你的后端 → 用 code 换 token
        ↓
    完成登录
```

### 10.4 你的项目适合用什么？

| 项目规模 | 推荐方案 | 理由 |
|---------|---------|------|
| 小型（学习/demo） | JWT 足够了 | 简单，好实现 |
| 中型（上线运营） | JWT + RBAC | 登录 + 权限分开 |
| 大型（多系统） | JWT + OAuth2.0 + RBAC | SSO 一个账号登录多个系统 |

## 十一、RBAC 角色权限控制

### 11.1 什么是 RBAC？

**RBAC = Role Based Access Control = 按角色分配权限**

```
不用 RBAC：
  张三能写文章 ✓
  张三能删文章 ✓
  张三能管理用户 ✓
  李四能写文章 ✓
  ...（每个人单独配置，烦死）
  王五离职了，要删他的权限 → 一个一个删，漏一个就是漏洞

用了 RBAC：
  定义角色：
    管理员 → 所有权限
    编辑 → 写文章、审稿
    普通用户 → 看文章、评论

  定义用户：
    张三 → 管理员
    李四 → 编辑
    王五 → 编辑

  王五离职 → 只用把他的角色删了 → 所有权限自动回收 ✓
```

### 11.2 三层结构

```
用户（张三）→ 角色（管理员）→ 权限（写、删、改、查）

用户（李四）→ 角色（编辑）→ 权限（写、审稿）

用户（赵六）→ 角色（普通用户）→ 权限（看）
```

### 11.3 数据库设计

RBAC 需要四张表：

- **权限表（permissions）**：定义每一种具体的权限，如 `article:create`、`article:delete`
- **角色表（roles）**：定义角色名称，如 `admin`、`editor`、`user`
- **角色-权限关联表（role_permissions）**：角色和权限的多对多关系
- **用户表（users）**：每个用户关联一个角色

### 11.4 与 JWT 的结合

在 JWT 的 Payload 中可以存放用户角色：

```json
{
  "user_id": 12345,
  "username": "张三",
  "role": "editor",
  "exp": 1709240200
}
```

这样每次请求时，服务器不仅知道"你是谁"，还能知道"你能做什么"。

## 十二、配置中心抽象

### 12.1 什么是"配置"？

**配置 = 会变的东西，但不应该写死在代码里**

```
反面例子：
  api_key = "sk-xxxx"  # 写死在代码里
  # 换 key 了？改代码 → 重新部署 → 烦死

正面例子：
  api_key = os.getenv("API_KEY")  # 写在环境变量里
  # 换 key 了？改 .env → 重启服务 → 搞定
```

### 12.2 你的项目有哪些配置？

**项目层面（Env 环境变量）**：
- Api Key — 第三方服务的钥匙
- LLM URL — 大模型调用的地址
- LLM 模型类型 — 用哪个模型（GPT-4 / GPT-3.5 / Claude）
- Mock 开关 — 测的时候不调真实 API，省钱
- Debug 开关 — 打开看详细日志

**业务层面（提示词）**：
- 用户提示词 — 写文章用的提示词、生成图片用的提示词
- 系统提示词 — AI 助手的角色设定、回复风格设定

### 12.3 方案一：集中式模块管理（最简单）

把提示词做成一个 Python 模块文件 `prompts.py`，和代码放在一起。

适合：**学习项目、小型项目、个人项目**

**优点**：
- 简单，改代码就行
- 不增加任何复杂度
- 一个人维护就够了

**缺点**：
- 改了提示词要改代码 → 要重新部署
- 没有版本记录
- 没法做 A/B 测试

### 12.4 方案二：存到数据库（做成系统）

把提示词存到数据库，做一个管理后台，运营和产品可以直接改提示词。

适合：**中型项目、有运营/产品团队参与**

数据库表设计：
- **prompts 表**：存储提示词名称、内容、版本号、是否启用
- **prompt_versions 表**：记录每个版本的历史，方便回滚

**优点**：
- 运行时可改（产品经理改完，用户刷新就生效）
- 可以做 A/B 测试（给不同用户用不同版本）
- 多版本管理，支持回滚

**缺点**：
- 复杂度高
- 增加数据库依赖

### 12.5 方案三：Nacos / Apollo（配置中心）

适合：**超大并发、多服务微服务架构**

```
大型互联网公司架构：
  用户服务 ──┐
  订单服务 ──┼── 100+ 台服务器
  支付服务 ──┤
  ...     ──┘

问题：改一个配置要登100台机器去改？不可能！

解决方案：配置中心
  ┌─────────────────────────────┐
  │        Nacos / Apollo        │
  │   （配置中心，只有一份）        │
  │                              │
  │   max_connections = 100      │
  │   api_timeout = 30s          │
  │   ...                         │
  └─────────────────────────────┘
            ↑ 实时同步
    ┌──────┴──────┬───────┬──────┐
    ↓            ↓       ↓      ↓
  用户服务     订单服务   支付服务  ...
```

**为什么一般不用**：
- 门槛高，适合大厂 Java 微服务团队
- 过度设计
- 国内 Java 圈用得多（Nacos=阿里，Apollo=携程）

### 12.6 三种方案对比

| 你的情况 | 推荐方案 | 理由 |
|---------|---------|------|
| 学习 / demo / 一个人开发 | 方案一 | 简单直接，够了 |
| 有产品/运营参与，要经常改提示词 | 方案二 | 运行时可改，不用每次都部署 |
| 日活 100 万+，几十台服务器，微服务架构 | 方案三 | 必须用配置中心 |
| 犹豫中 | 方案一先上 | 先跑起来，后面业务复杂了再升级 |

## 十三、健康检查与容器化部署

### 13.1 为什么需要健康检查？

```
你的服务部署在云服务器上
      ↓
半夜 3 点服务器挂了
      ↓
没有健康检查 → 没人知道 → 用户用不了 → 投诉 → 损失
      ↓
有健康检查 → 系统自动检测到挂了 → 立即报警 → 工程师爬起来修 → 影响最小化
```

健康检查 = **定时问"你还活着吗"**

### 13.2 健康检查是怎么工作的？

```
监控程序（轮询）
    ↓
每 30 秒访问一次 /health 接口
    ↓
接口返回 200 OK + {"status": "healthy"}
    ↓
好，下一轮
    ↓
接口返回 500 / 超时不响应
    ↓
判定：服务挂了
    ↓
报警！
```

### 13.3 健康检查的两个层次

**基础健康检查（/health）**：只返回 "healthy" 或 "unhealthy"，监控程序用它判断服务是否存活。

**详细健康检查（/health/detailed）**：检查数据库、Redis、LLM 服务等所有依赖的连通性，返回每个依赖的状态。工程师用它定位具体哪里出了问题。

### 13.4 报警通知方式

| 方式 | 场景 | 体验 |
|------|------|------|
| 邮件 | 不紧急，事后看 | 慢，可能不看 |
| 短信 | 紧急 | 快，但可能被当垃圾短信 |
| 电话 | 重大故障 | 最快，但打扰大 |
| 钉钉/飞书/企业微信群 | 最常用 | 即时 + 有上下文 + 免费 |

### 13.5 Docker 是什么？

**Docker = 把你的程序和它需要的环境打包成一个"集装箱"**

```
传统部署：
  代码在 Windows 上跑得起来
  换到 Linux 服务器 → 挂了（环境不一样）
  原因：Python 版本不同、缺少某个库、系统路径不同...

用 Docker：
  把代码 + Python环境 + 依赖库 + 配置文件 全部打包
      ↓
  变成一个 Docker 镜像
      ↓
  无论在哪台服务器 → 运行这个镜像 → 都能正常跑
```

**生活类比**：
```
传统部署：
  厨师带着食材去你家做饭
  → 结果你家没锅 → 做不了

Docker：
  厨师带着食材 + 便携锅 一起过去
  → 随时都能做 → 味道一样
```

### 13.6 K8s（Kubernetes）是什么？

**K8s = Docker 集群的"指挥官"**

```
没有 K8s（手动管 Docker）：
  你有 10 台服务器
      ↓
  这台挂了？手动去另一台启动
      ↓
  用户量暴增？手动把服务复制到另外 5 台
      ↓
  要更新版本？一台一台手动更新
      ↓
  运维工程师要累死了

有 K8s：
  你告诉 K8s："我要跑 3 个 Python 服务，自动管理"
      ↓
  K8s 自动分配到 10 台服务器里
      ↓
  某一台挂了？K8s 自动把服务挪到其他机器上 ✓
      ↓
  用户暴增？K8s 自动扩容到 10 个服务 ✓
      ↓
  要更新？K8s 滚动更新，一台一台换，不影响用户 ✓
```

### 13.7 探针（Probe）

K8s 会定时"戳"你的服务，问你还活着吗。两种探针：

```
存活探针（Liveness Probe）：
  问：你还活着吗？
  回答：
    - 200 → 活着，继续跑
    - 503 / 超时 → 死了，我帮你重启
  
  场景：服务没死，但卡住了（比如死循环）

就绪探针（Readiness Probe）：
  问：你准备好了吗？可以接收用户请求了吗？
  回答：
    - 200 → 准备好了，来流量吧
    - 503 → 还没准备好，先别给我发请求
  
  场景：服务启动了，但还在预热（比如加载大模型）

启动探针（Startup Probe）：
  问：你启动了吗？
  场景：服务启动很慢（比如要加载 5GB 模型）
```

### 13.8 完整流程图

```
用户请求
    ↓
K8s 收到请求
    ↓
K8s 问 /ready（Readiness Probe）→ 通了 → 转发给服务
    ↓
服务处理请求
    ↓
同时 K8s 定时问 /health（Liveness Probe）
    ↓
    ┌─ 正常 → 继续跑
    ↓
    └─ 挂了 / 超时 → K8s 自动重启这个容器
    ↓
重启后 K8s 问 /ready
    ↓
通了 → 恢复接收流量
    ↓
同时监控系统记录所有 /health 的历史
    ↓
发现趋势（比如健康检查时间越来越长）
    ↓
提前报警："服务快不行了，提前处理"
```

## 十四、限流与熔断

### 14.1 限流（Rate Limiting）

#### 限流是什么？

```
食堂窗口：
  平时 → 100人吃饭 → 食堂没问题
  中午高峰期 → 1000人冲进来 → 厨房炸了 → 所有人吃不上

限流 = 在门口放个闸机 → 每分钟放 100 人进去 → 厨房稳了
```

限流保护的是**你自己**，不被海量的请求压垮。

#### 限流维度

| 维度 | 例子 | 场景 |
|------|------|------|
| 按 IP | 同一个 IP 每分钟最多 60 次 | 防爬虫、恶意请求 |
| 按用户 | 登录用户每分钟最多 30 次 | 防滥用 |
| 按接口 | `/generate` 每分钟最多 10 次 | 保护重资源接口 |

### 14.2 熔断（Circuit Breaker）

#### 熔断是什么？

```
正常情况：
  你的服务 → 调 LLM API → 成功 → 返回结果

LLM 开始抽风：
  你的服务 → 调 LLM API → 超时 10 秒 → 失败
  你的服务 → 调 LLM API → 超时 10 秒 → 失败
  你的服务 → 调 LLM API → 超时 10 秒 → 失败

问题：每个请求都等 10 秒才失败 → 100个用户 × 10秒 = 1000秒浪费

熔断：
  连续失败 3 次 → 熔断器打开
      ↓
  你的服务 → 熔断器直接拦截 → "LLM 挂了，走备用" → 立刻返回
      ↓
  不等了！用户秒拿到结果
```

熔断保护的是**你自己**，不被下游服务拖死。

#### 熔断的三种状态

```
┌─────────┐    连续失败5次     ┌─────────┐    30秒后     ┌──────────┐
│ 关闭状态 │ ──────────────▶ │ 打开状态 │ ──────────▶ │ 半开状态  │
│ (正常)  │                  │ (熔断中) │              │ (试探)   │
└─────────┘                  └─────────┘              └──────────┘
     ↑                              │                       │
     │       成功 ◀─────────────────┘                       │
     │                                                      │
     └──────────────────────────────────────────────────────┘
                               失败 → 继续打开
```

- **关闭状态**：正常状态，所有请求都放行。失败计数达到阈值后自动转到打开状态。
- **打开状态**：熔断中，所有请求不再尝试主服务，直接走备用方案。一段时间后转到半开状态。
- **半开状态**：试探性地放行一个请求。如果成功，恢复关闭状态；如果失败，继续打开。

### 14.3 限流 vs 熔断（核心区别）

```
┌─────────────────────────────────────────────────────────┐
│  限流：保护自己（不让自己被用户压垮）                      │
│  触发条件：请求量超过阈值                                │
│                                                         │
│  熔断：保护自己（不被下游服务拖死）                      │
│  触发条件：下游连续失败/超时                             │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  限流例子：                                             │
│  用户 1 秒内发了 100 个请求                              │
│  → 限流拦截 → "你太快了，歇会儿"                        │
│                                                         │
│  熔断例子：                                             │
│  LLM API 超时了 5 次                                   │
│  → 熔断打开 → 不再试 LLM → 直接返回降级结果             │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## 十五、降级策略与错误边界

### 15.1 什么是降级？

```
主服务挂了（LLM 不可用）
    ↓
你等它吗？→ 等 10 秒 → 超时 → 用户骂你
    ↓
不等了 → 直接返回兜底方案 → 用户至少能拿到东西
```

**降级 ≠ 服务挂了**
**降级 = 走备用路线，慢一点，但能到**

**生活类比**：
```
高速封了
    ↓
你：
  A. 在收费站在那等高速解封 → 等了 2 小时 → 到了但很累
  B. 走国道 → 多花了 1 小时 → 但到了

  选 B = 降级
```

### 15.2 简单降级 vs 熔断降级

```
┌────────────────────────────────────────────────────────────────┐
│   【简单降级】                                                  │
│                                                                │
│   请求 ──▶ 主服务(每次都试) ──▶ 失败 ──▶ 备用服务               │
│              │                                                 │
│              │ 超时10秒                                        │
│              ▼                                                 │
│         用户等了10秒才拿到结果                                  │
│                                                                │
│   问题：主服务挂了，每个用户都要等10秒                           │
│                                                                │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│   【熔断降级】                                                  │
│                                                                │
│   请求 ──▶ 熔断器检查 ──▶ 开着 ──▶ 直接备用服务                 │
│              │                                                 │
│              │ 0秒                                             │
│              ▼                                                 │
│         用户秒拿到结果                                          │
│                                                                │
│   好处：快速失败，不浪费时间                                     │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 15.3 降级策略分层

```
┌──────────────────────────────────────────────────────────┐
│                                                          │
│  第1层：主服务                                           │
│  → 质量最高，效果最好                                     │
│                                                          │
│  第2层：备用服务                                          │
│  → 质量稍差，但能用                                       │
│                                                          │
│  第3层：兜底文案                                         │
│  → 质量一般，但能返回，不让用户白等                       │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 15.4 完整降级流程

```
请求1 ──▶ 尝试主服务 ──▶ 失败(等了10秒) ──▶ 走备用 ──▶ 失败计数=1
请求2 ──▶ 尝试主服务 ──▶ 失败(等了10秒) ──▶ 走备用 ──▶ 失败计数=2
请求3 ──▶ 尝试主服务 ──▶ 失败(等了10秒) ──▶ 走备用 ──▶ 失败计数=3 ──▶ 触发熔断！
────────────────────────────────────────────────────────────────
请求4 ──▶ 熔断器拦截 ──▶ 直接走备用（0秒）
请求5 ──▶ 熔断器拦截 ──▶ 直接走备用（0秒）
请求6 ──▶ 熔断器拦截 ──▶ 直接走备用（0秒）
...
30秒内都不尝试主服务
────────────────────────────────────────────────────────────────
30秒后 ──▶ 半开状态 ──▶ 放一个请求试试 ──▶ 成功了！──▶ 恢复正常
```

## 十六、用户反馈闭环

### 16.1 反馈闭环是什么？

```
用户用了你的服务
    ↓
你觉得效果怎么样？（收集反馈）
    ↓
这些反馈 → 变成训练数据 → 优化提示词
    ↓
效果变好了 → 用户更满意 → 更多人用
    ↓
继续收集反馈 → 继续优化 → 正向循环
```

### 16.2 为什么要做反馈闭环？

```
没有反馈闭环：
  上线了提示词 A
  → 不知道好不好（用户骂了也不知道为什么）
  → 改提示词 B → 不知道改好了没有 → 瞎改

有反馈闭环：
  上线了提示词 A → 收集用户评价 → 评分 3.2/5
  → 分析问题：用户说"太正式了"
  → 改提示词 B → 再上线 → 评分 3.8/5
  → 确实改进了！有数据支撑
```

### 16.3 数据库设计

**generation_tasks 表**：每次生成任务，记录下来
- `id` — 任务 ID
- `user_id` — 哪个用户
- `task_type` — 任务类型（'article' / 'image' / 'summary'）
- `prompt_version` — 用的哪个提示词版本（'v1.0'）
- `input_text` — 输入
- `output_text` — 输出
- `created_at` — 时间

**user_feedbacks 表**：用户反馈
- `id` — 反馈 ID
- `task_id` — 关联的任务
- `rating` — 1-5 星评分
- `thumbs_up` / `thumbs_down` — 点赞/踩
- `comment` — 用户写的评价文字

### 16.4 提示词评测集

每次改提示词之前，先用固定的测试用例跑一遍，看分数有没有提升：

```
评测用例1：主题="夏天防晒技巧"，期望风格="亲切、有趣"，最低字数=200
评测用例2：主题="职场穿搭建议"，期望风格="专业、有干货"，最低字数=300
评测用例3：主题="周末遛娃好去处"，期望风格="温馨、实用"，最低字数=250

每次改动提示词 → 跑完这3个用例 → 计算平均分 → 对比上次
```

### 16.5 A/B 测试

```
提示词A（方案A）：→ 给 Android 用户用
提示词B（方案B）：→ 给 iOS 用户用
运营看数据 → 哪个转化率高 → 全量切过去

举例：
  v1.0："请写一篇有趣的文章"  → 点击率 3%
  v1.1："请写一篇让人忍不住点赞的文章" → 点击率 8%
  → 全量切到 v1.1，多了 5% 点击率
```

## 十七、多层防护体系

### 17.1 真实企业完整方案

```
┌─────────────────────────────────────────────────────────┐
│                    多层防护体系                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   第1层：CDN/WAF                                        │
│   └── DDoS防护、恶意IP封禁                               │
│                                                         │
│   第2层：API网关（Nginx/Kong/AWS）                       │
│   └── 全局限流（按IP、按用户、按接口）                    │
│                                                         │
│   第3层：应用层（代码）                                  │
│   └── 业务级限流（VIP用户配额更高）                      │
│   └── 熔断器（下游服务保护）                             │
│                                                         │
│   第4层：下游服务                                        │
│   └── LLM服务自己也有限流（火山引擎API限制）              │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 17.2 每层防护详解

**第1层：CDN/WAF**
- CDN：把静态内容放到全国各地的节点，用户就近访问
- WAF：Web应用防火墙，过滤 SQL 注入、XSS 等攻击

**第2层：API 网关**
- 统一认证：所有请求先过网关，验证 Token
- 全局限流：按 IP、用户、接口维度限制请求频率
- 统一日志：所有请求留下记录

**第3层：应用层**
- 业务级限流：VIP 用户配额更高，普通用户配额更低
- 熔断器：当下游服务异常时，快速失败
- 降级：主服务挂了走备用方案

**第4层：下游服务**
- LLM 服务商自己也有 QPS 限制
- 数据库也有最大连接数限制

## 十八、面试话术汇总

### JWT 鉴权

> "我在项目中实现了基于 JWT 的无状态认证方案。用户登录时，后端验证用户名密码后，使用 HS256 算法签发包含 user_id 和 role 的 Token，有效期 24 小时。所有受保护的接口通过 FastAPI 依赖注入验证 Token，全程不查数据库。配合 Argon2 做密码哈希，比 BCrypt 更安全。针对 Token 撤回的需求，结合 Redis 维护了一个短 token 黑名单列表。"

### OAuth2.0

> "我了解 OAuth2.0 的四种授权模式。对于本项目的场景，使用授权码模式对接微信登录，用户点击微信登录后跳转到授权页，授权后回调后端换取 Token 完成登录。SSO 能让用户一个账号登录多个相关系统，避免重复注册。"

### RBAC 权限控制

> "我在项目中设计了 RBAC 权限体系。数据库有四张表：用户表、角色表、权限表、角色权限关联表。每个用户关联一个角色，每个角色有多个权限。管理员拥有所有权限，编辑可以写文章和审稿，普通用户只能浏览。离职时删除用户角色即可自动回收所有权限，不会遗漏。"

### 配置中心

> "我在项目中做了两层配置抽象。第一层是环境变量层，放 API Key、模型类型这类技术配置，配合 Pydantic 做类型校验。第二层是业务配置层，把提示词模板单独抽象成模块，支持热加载。这样产品和运营要改提示词的时候，不用找工程师改代码，也不用重新部署服务。后续如果业务规模上去，可以把提示词搬到数据库，支持版本管理和 A/B 测试。"

### 健康检查与容器化

> "我在项目里实现了完整的健康检查体系。后端服务暴露了 `/health` 接口，探测数据库、Redis 的连通性。服务部署在 Docker 容器里，由 K8s 管理容器生命周期，通过 Liveness Probe 做存活检测、Readiness Probe 做就绪检测。监控数据上报到公司的 Prometheus + Grafana 平台，配合 AlertManager 配置了钉钉群报警，一旦服务不健康，5 分钟内就能触达值班工程师，实现故障的早发现、早处理。"

### 限流与熔断

> "我在项目中实现了多层限流和熔断保护。网关层用 Nginx 做了全局限流，按 IP 和接口维度限制请求频率，防止突发流量冲垮服务。在应用层针对 LLM 调用加了熔断器，当下游服务连续失败 5 次后自动打开熔断，直接走降级方案，不再等待超时的请求。结合降级策略，熔断期间会返回备用内容，保证服务可用。实测在 LLM 服务抖动期间，接口 P99 延迟从 10 秒降到了 200 毫秒。"

### 降级策略

> "我实现了多层降级策略。当 LLM 服务不可用时，第一层尝试备用 LLM，第二层返回静态兜底文案，保证接口永远能返回数据，不让用户白等。通过熔断器控制降级时机，熔断打开后不再尝试主服务，直接走备用方案，将 P99 延迟从 10 秒降到 200 毫秒。降级结果会打上 is_degraded 标记，前端据此展示不同的 UI 状态。"

### 反馈闭环

> "我设计了完整的用户反馈闭环。每次生成任务结束后让用户打分 1-5 星，收集真实使用数据。反馈数据积累成评测集，每次改提示词前先跑一遍评测集，看平均分有没有提升，形成数据驱动的优化闭环。"

## 附录：概念总结

| 概念 | 是什么 | 解决什么问题 |
|------|--------|------------|
| LangGraph | 用图的形式描述业务流程 | 让复杂的多步骤 AI 工作流易于维护 |
| interrupt | 暂停工作流等待外部信号 | 实现人在回路的 AI 工作流 |
| Checkpointer | 状态持久化 | 断点续传、故障恢复、多线程并行 |
| JWT | 把用户信息加密成一张"电子身份证" | 不用每次查数据库，省资源 |
| OAuth2.0 | 一套"让别人帮你登录"的协议 | 单点登录、多系统共享账号 |
| RBAC | 按角色分配权限的管理方式 | 不用给每个人单独配权限 |
| 配置中心 | 把配置集中管理 | 改配置不用重新部署 |
| 健康检查 | 定时问"你还活着吗" | 故障早发现 |
| Docker | 把程序打包成集装箱 | 在哪都能跑 |
| K8s | Docker 集群的指挥官 | 自动管理、自动扩容 |
| 探针 | K8s 定时戳服务问状态 | 存活检测 + 就绪检测 |
| 限流 | 控制请求速度 | 不被流量冲垮 |
| 熔断 | 下游挂了就不等了 | 快速失败，省资源 |
| 降级 | 主服务挂了走备用 | 让用户至少能用到东西 |
| 反馈闭环 | 用户反馈 → 优化 → 验证 | 持续改进，数据驱动 |
| 评测集 | 固定的测试用例库 | 每次改提示词有量化依据 |
| A/B 测试 | 两套方案同时跑，对比效果 | 科学决策，不是拍脑袋 |
